Why SEPF

Business
  • Predictable delivery through consistent patterns and responsibilities.
  • Lower total cost of ownership via shared components and fewer regressions.
  • Faster onboarding — developers navigate familiar structures across orgs.
Technical
  • DatabaseContext centralizes access, improves performance, and clarifies ownership.
  • Selectors replace ad‑hoc SOQL; enable method reuse per context without duplication.
  • UoW (Unit of Work) orchestrates DML with fewer triggers and cleaner transactions.

What you’ll get

FFLIB & Enterprise Design Patterns

SEPF — origin & highlights

A pragmatic evolution of FFLIB with performance and maintainability front and center.

Metadata driven

Leverage metadata for configuration over code and consistent rules across contexts.

DatabaseContext

Centralized access: performance, selector test classes, clear class responsibility.

Selector usage

Use DatabaseContext; avoid enforceFLS overhead, reduce duplicates, enable method reuse per UI/API/batch.

UoW usage

Transactional orchestration for DML with fewer triggers and cleaner commits.

Selector simplification

Small, purposeful methods; no cross‑cutting validation inside selectors.

Trigger Framework

Deterministic order, bulk-safe, minimal branching; per‑object handlers.

Unit Test Framework

Factories, selector stubs, context fixtures for fast, focused tests.
